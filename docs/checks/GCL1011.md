# GCL1011 — double-lock

> A second Lock() is taken while the first is still held.

|           |                              |
|-----------|------------------------------|
| Code      | `GCL1011` |
| Slug      | `double-lock` |
| Primitive | `sync.Mutex`, `sync.RWMutex` |

## Why it matters

Re-locking a non-reentrant sync.Mutex that is already held deadlocks the goroutine immediately.

## Examples

The linter flags code like this:

```go
mu.Lock()
mu.Lock() // second lock on a held, non-reentrant mutex deadlocks
defer mu.Unlock()
```

Write it like this instead:

```go
mu.Lock()
defer mu.Unlock()
// no second Lock on the same mutex
```

## Suppressing this check

Add an inline directive on the offending line. Either the canonical code or the legacy slug is accepted:

```go
foo() // goconcurrencylint:ignore GCL1011
foo() // goconcurrencylint:ignore double-lock
```
