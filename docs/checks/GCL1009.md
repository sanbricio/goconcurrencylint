# GCL1009 — goroutine-lock-deadlock

> A goroutine started while a lock is held tries to take the same lock before the parent releases it.

|           |                              |
|-----------|------------------------------|
| Code      | `GCL1009` |
| Slug      | `goroutine-lock-deadlock` |
| Primitive | `sync.Mutex`, `sync.RWMutex` |

## Why it matters

The child blocks on a lock the parent still holds while the parent waits on the child — a guaranteed deadlock.

## Examples

The linter flags code like this:

```go
mu.Lock()
go func() {
	mu.Lock() // blocks on the parent, which is waiting on this goroutine
	defer mu.Unlock()
}()
// parent keeps holding mu
```

Write it like this instead:

```go
mu.Lock()
// ... use the shared state
mu.Unlock()
go func() {
	mu.Lock()
	defer mu.Unlock()
}()
```

## Suppressing this check

Add an inline directive on the offending line. Either the canonical code or the legacy slug is accepted:

```go
foo() // goconcurrencylint:ignore GCL1009
foo() // goconcurrencylint:ignore goroutine-lock-deadlock
```
