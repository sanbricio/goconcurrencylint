# GCL1001 — lock-without-unlock

> A Lock()/RLock() call has no matching Unlock()/RUnlock() on some execution path.

|           |                              |
|-----------|------------------------------|
| Code      | `GCL1001` |
| Slug      | `lock-without-unlock` |
| Primitive | `sync.Mutex`, `sync.RWMutex` |

## Why it matters

The mutex stays locked, so every later Lock() on it blocks forever — a deadlock that usually surfaces only under load.

## Examples

The linter flags code like this:

```go
func update(mu *sync.Mutex, data map[string]int) {
	mu.Lock()
	data["n"]++ // returns with the mutex still locked
}
```

Write it like this instead:

```go
func update(mu *sync.Mutex, data map[string]int) {
	mu.Lock()
	defer mu.Unlock()
	data["n"]++
}
```

## Suppressing this check

Add an inline directive on the offending line. Either the canonical code or the legacy slug is accepted:

```go
foo() // goconcurrencylint:ignore GCL1001
foo() // goconcurrencylint:ignore lock-without-unlock
```
